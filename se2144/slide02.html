Phase I: The Foundation

Setting up the environment and executing your first NextJS application with industry-standard patterns.

Modern Installation

We recommend Yarn for its speed, network performance, and reliability. However, NPM remains a robust fallback.

Using Yarn


yarn create next-app --typescript

Using NPM


npx create-next-app@latest --typescript

Local Development Scripts

Ensure your package.json contains these vital lifecycle scripts.

  • "dev": Starts the development server with Hot Module Replacement (HMR).
  • "build": Compiles and optimizes the application for production.
  • "start": Runs the production build generated by the build script.
Run yarn dev in your root directory. Your application will typically be accessible at: http://localhost:3000

Folder Structure: The "Pages" Core

NextJS utilizes a file-system based router built on the concept of pages.

  • The basic element is a React Component exported from a file within the pages/ directory.
  • The default entry point is always pages/index.tsx.
  • Every new file in this folder automatically becomes a URL endpoint. Zero manual routing configuration required.

Phase II: Power Tools

Mastering configuration, TypeScript integration, and environment management.

Config Customization

The next.config.js file allows you to inject custom logic or third-party plugins directly into the Webpack build process.


const nextConfig = {
  reactStrictMode: true, // Highlights potential problems
  webpack: (config, { buildId, dev, isServer, webpack }) => {
    // Inject custom webpack configurations here
    return config;
  }
}
module.exports = nextConfig

Managing Secrets

⚠️ Prevent Leak Vulnerabilities

  • Never hardcode API keys, database credentials, or secrets in your components.
  • Use .env.local for local development environment variables.
  • Access these values in your server-side logic using process.env.SECRET_NAME.

Note: Any modification to your environment files requires a full development server restart.

TypeScript Workflow

NextJS simplifies TypeScript implementation drastically.

  1. Create an empty tsconfig.json file in your root.
  2. Run yarn dev.
  3. NextJS automatically populates the config file with strict defaults and generates a next-env.d.ts file.
Rule of Thumb: Use .ts or .tsx for implementation (logic and components). Use .d.ts strictly for ambient type declarations.

Phase III: Styling

Crafting the visual layer with scoped styles and modern tools.

Advanced Styling Approaches

  • SCSS Modules: Native support. Name files as [name].module.scss. Styles are locally scoped by default, preventing global namespace collisions.
  • Global Styles: Reserved exclusively for pages/_app.tsx. Do not import global CSS anywhere else.
  • Tailwind CSS: (Our T3 Stack choice). Utility-first CSS framework that scales seamlessly with NextJS component architecture.

Phase IV: Routing

Mastering dynamic URL segments, programmatic navigation, and shallow routing.

Routing Mechanisms

Type File Naming Description
Static about.tsx Maps directly to exactly /about
Dynamic [recipe].tsx Captures URL variables (e.g., /pancakes) into the router query object.
Catch-all [...slug].tsx Captures infinite nested paths (e.g., /docs/v1/auth) as an array.

Shallow Routing

Allows you to change the URL without re-running data fetching methods (like getServerSideProps).


router.push('/?page=2', undefined, { shallow: true })
Optimization Use Case: Perfect for pagination, filtering, or tab switching where the page state changes but the heavy underlying data core does not need a complete server re-fetch.
