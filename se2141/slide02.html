Traditional File Systems

Before DBMSs, corporate records were stored in separated operating system text or custom binary files.

  • Data was kept in flat structures handled by customized software programs written by different teams.
  • As businesses grew, this application-centric storage method quickly hit major architectural roadblocks.

File System Disadvantages

  • Data Redundancy & Inconsistency: The same information is duplicated across different file paths, leading to mismatched fields.
  • Difficulty in Accessing Data: Requires writing completely new code scripts just to pull simple customized reports.
  • Data Isolation: Records are scattered in different formats, making integration highly difficult.

Atomicity & Concurrency Disadvantages

  • Atomicity Violations: If a system crashes mid-transfer, money could leave Account A without ever arriving in Account B.
  • Concurrent-Access Anomalies: Multiple updates running at the exact same time can overwrite each other's changes.
  • Security Problems: Hard to enforce granular read/write access permissions on raw text files.

The Solution: The DBMS

A Database Management System (DBMS) consists of a collection of interrelated data and a set of software programs to access that data safely and efficiently.

  • Provides a centralized repository that solves data isolation and redundancy.
  • Ensures highly reliable data storage despite hardware failures, power cuts, or thousands of concurrent user requests.

Levels of Data Abstraction

To hide structural complexity from developers, a DBMS abstracts data into three distinct architectural layers:

  • Physical Level: The lowest tier. Describes *how* the records are actually stored in bytes, tracks, or sector indexes on disks/SSDs.
  • Logical Level: The intermediate tier. Describes *what* data is stored in the database, and what relationships exist among those data points. (e.g., tables, columns, constraints).
  • View Level: The highest tier. Hides details of data types for security or simplicity, exposing only customized entry forms or user specific dashboards.

Schemas and Instances

Understanding the difference between structural design and real-time state:

  • Database Schema: The overall logical design framework of the database. This rarely changes (analogous to variable declarations or class types in programming).
  • Database Instance: The actual snapshot collection of data stored in the database at a specific moment in time (analogous to the values assigned to variables at runtime).

Data Modeling Blueprints

A data model is a collection of conceptual tools for describing data, data relationships, data semantics, and consistency constraints.

  • Relational Model: Represents data in tables (relations) with rows and columns. This is the dominant industry standard model.
  • Entity-Relationship Model (ERM): A graphical model widely used for database design layout analysis.
  • Semi-Structured Model (JSON/XML): Allows individual data items to have different sets of attributes.

Database Languages

We interact with a database engine using two specialized categories of instructions within SQL:

  • Data Definition Language (DDL): Used to specify the database schema structure, tables, types, and constraints.
    CREATE TABLE students (id INT PRIMARY KEY, name VARCHAR(50));
  • Data Manipulation Language (DML): Used to express database queries and updates (inserting, deleting, updating data rows).
    SELECT name FROM students WHERE id = 101;

Chapter 1 Recap

  1. File Systems Fail: Duplication, lack of concurrency control, and lack of atomicity make flat file systems unviable for scalable enterprise data applications.
  2. DBMS Abstract Complexity: By isolating physical layouts from logical models, databases allow developers to build flexible systems without worrying about low-level disk management.
  3. Data Models Form Blueprints: Relational structures and structured query languages (SQL) guarantee standard, robust ways to store and read data.
